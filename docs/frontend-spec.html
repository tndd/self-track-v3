self-track-v3 フロントエンド仕様書

画面モックアップと仕様定義が同居した生きたインタラクティブ仕様書

📝 全体概要 & パッシブ・トラッキング設計

本アプリケーションは、「ユーザーが行った能動的アクション」と「身体・精神的なコンディションの変化」の因果関係を、端末内（オンデバイス）で統計的に特定するための体調記録・分析アプリです。

💡 設計思想: 例外記録 (パッシブ・トラッキング)
ユーザーに入力を強制しない設計を採用しています。「体調が良い時・悪い時」「特定の薬を飲んだ時」など、イベントが発生した際のみ記録します。未評価の時間は自動的に平常状態 (0) として扱われます。

本インタラクティブ仕様書の使いかた

左側のスマートフォンモックアップ内のメニュー(☰)から画面を切り替えると、この仕様書の表示セクションが現在操作中の画面に同期し、該当する画面状態やデータ構造の仕様がハイライト表示されます。

左側の画面は、JavaScriptで動くインメモリSQLite風DBで実際にCRUD操作が機能します。新しいタグの追加、レコードの記録、レコードの長押し削除などを行うと、カレンダーや統計の各数値・グラフに即座にフィードバックされます。ぜひお試しください。

📱 Track（体調記録画面）仕様 M3 準拠

体調値 (5段階)、コメント、及びその瞬間に紐付く複数の行動/症状/環境タグを瞬時に記録・可視化するためのメイン画面です。

画面が持ち得る状態 (State)

状態名 説明 UI表示と遷移
通常タイムライン状態 当日の記録を時系列降順で並べる。 コメント、体調バッジ、付随タグを表示。
Composer展開時 (Status) 下部の + ボタン、またはテキスト欄タップで展開。 体調値 (1〜5) を選択する最小ピルエリア。
Tag展開時 (Composer拡張) Composer内「タグを追加」を押下して展開。 最近使用したタグ及びグループ別アコーディオンが表示され、複数選択可能。
長押しアクションメニュー タイムラインの記録を長押ししてボトムシート表示。 誤操作防止。編集・削除を選択可能にする。

データとの紐付け

Composerで「記録する」を押した際、Drift を用いて以下のデータベース書き込みトランザクションが実行されます。

// Recordsテーブルへの新規インサート final recordId = uuid.v4(); await db.recordsDao.insertRecord( RecordsCompanion.insert( id: recordId, timestamp: DateTime.now(), value: selectedValue, // -2 〜 2 の値 comment: Value(commentText), ) ); // RecordTags（中間テーブル）への一括紐付け for (var tagId in selectedTagIds) { await db.recordsDao.insertRecordTag( RecordTagsCompanion.insert( recordId: recordId, tagId: tagId, value: const Value(1.0), // v1.0 では 1.0 固定 ) ); }
⚠️ 注意: 物理削除の連動 (CASCADE)
レコードを削除した際、関連する record_tags 中間データは外部キー制約 ON DELETE CASCADE によって自動的に削除されます。Tagsマスタは削除されずそのまま保持されます。

📅 Calendar（カレンダー画面）仕様 M5 準拠

月単位で体調スコアの変化を大局的に捉えるための画面です。

1. 日次スコアの5段階丸めと多色リング

  • 単一レコードの日: そのレコードの体調値 (-2〜2) を 1〜5 のカラーでセルに塗りつぶします。
  • 複数レコードの日 / 積分計算日: 1日の平均体調値を台形公式 (AUC) で求め、その値を四捨五入して色を決定します。
  • 多色リング (Ring): 1日に異なる属性のタグや大きく異なる体調の記録が複数ある場合、UIの表現力向上として、Conic Gradient を用いて周囲に複数の体調色を環状にグラデーション描画します（案3）。

2. 「今月の割合」ドーナツチャート

月次内の各体調スコア (1〜5) の日数をカウントし、その比率をSVGリングで描画します。中央には、その月のAUC加重平均値 (UI表現: 1.0〜5.0) を表示します。

3. 7日間の傾向 (スプライン曲線 & 基準線ピボット塗り)

直近7日の体調積分値の推移を表示します。中央の点線が「平常 (3=普通)」となっており、それより上の「良い・最高」状態は青〜緑のグラデーション、下の「悪い・最悪」状態はオレンジ〜赤のグラデーションで塗りつぶすことで、一目で今週のコンディションの良し悪しが感覚的に理解できるようにしています。

🧮 台形公式 (Trapezoidal Rule) による日次積分

1日の体調のトータル評価値は、単純な算術平均ではなく、時間差を考慮した台形積分 (AUC: Area Under the Curve) によって算出されます。

// JSによる簡易再現コード function getDailyAUC(points) { let area = 0; for (let i = 0; i < points.length - 1; i++) { let dt = (points[i+1].t - points[i].t) / (24 * 3600 * 1000); // 1日を1.0とした時間比 let valAvg = (points[i].val + points[i+1].val) / 2; area += valAvg * dt; } return area; // -2.0 〜 2.0 の連続値 }

📊 Stats（統計分析画面）仕様 M6 準拠

蓄積された時系列ログから、因果関係を示唆するシグナルを抽出して提示する画面です。

1. 12時間減衰 (Decay) アルゴリズム

自己評価が記録されない「平常時」のデータを正しく扱うため、12時間以上記録がない期間には「値0 (平常) への収束仮想ポイント」をメモリ上で仮想的に挿入し、積分や分析の偏りを防ぎます。

2. イベントロック平均 (Event-Locked Average)

特定のタグ（例: 「コーヒー」や「頭痛」）が発生した時間（T=0）を起点として、前後12時間における体調値の変動平均をスプライン曲線でプロットします。 「薬を服用した後に体調がどう戻っていくか」「睡眠不足のあとにどれくらい体調が悪化するか」などの時系列寄与を視覚的に特定できます。

3. 関連分析 (オッズ比 / Fisherの正確確率検定 / リフト値)

「行動タグ（x）」と「症状タグ（y）」が同日に発生する共起関係を2×2分割表を用いて評価します。

症状あり (y) 症状なし (y)
行動あり (x) a (共起日) b
行動なし (x) c d
  • リフト値: P(症状|行動) / P(症状)。基準の1.0より高ければ、その行動をした日に症状が起きやすい傾向を示します。
  • Fisher p値: サンプル数（N数）が極めて少ない個人データ環境に合わせ、漸近的検定（カイ二乗等）ではなく、周辺和固定の超幾何分布に基づいたFisherの正確確率検定を自前実装してp値を計算し、信頼性を評価します。
📝 データ不足のハンドリング
行動または症状の発生回数が合計3回未満の場合は、誤った解釈を防ぐため「データ不足」として計算値（リフト値）をマスクし、リスト下部に配置します。

🏷️ Tags（タグ管理画面）仕様 M2 準拠

記録に使用するタグを管理する画面です。タグは「行動」「症状」「要因・環境」といったグループごとに分類されます。

物理削除を廃した「アーカイブ」構造

ユーザーが誤ってタグを消した際、過去の記録（record_tags）の整合性を失わせないため、タグは物理削除ではなく論理削除（アーカイブ）処理を行います。

  • アーカイブ状態のタグ (isArchived = true):
    • 新規記録画面 (Composer) のタグサジェスト候補には出現しません。
    • 過去のタイムライン履歴や、カレンダー、統計画面には引き続きタグデータが残って評価対象となります。
    • タグ管理画面の「アーカイブ済み」セクションからいつでも有効化（アーカイブ解除）に戻せます。
// Tags テーブル定義 (Drift) class Tags extends Table { TextColumn get id => text()(); TextColumn get name => text().unique()(); TextColumn get group => text().named('tag_group')(); // SQLiteの予約語回避のため BoolColumn get isArchived => boolean().withDefault(const Constant(false))(); @override Set get primaryKey => {id}; }

⚙️ Settings（設定画面）仕様 M7 準拠

アプリ全体の管理機能を提供します。v1.0 においては安全性を最優先した「全データの消去」のみを実装します。

全データ消去フロー

誤操作によるデータ消失を防ぐため、以下の2段階のモーダル確認フローを踏んでローカルDBをクリアします。

  1. 「全データの消去」ボタンをタップ。
  2. 警告モーダルが表示され、「本当に削除しますか？」に対し赤文字の「削除実行」をタップして初めてDB全レコード消去のトランザクションが走ります。

データ消去完了後、すべての画面がデータなしの「空状態（Empty State）」に切り替わり、クラッシュが発生しないように制御します。
